Overview

A dystopian digital archive

Project 451 is a speculative fiction web experience inspired by Fahrenheit 451 — Ray Bradbury's novel about a future where books are banned and "firemen" burn them. The project transforms the novel's themes of censorship, memory, and resistance into an interactive digital archive.

The concept: what if the book-preservers in the novel had a secret digital archive? Users enter a classified, terminal-style interface and navigate through censored documents, hidden records, and narrative-driven content — as if they're accessing something they're not supposed to see.

Design Decisions

When the interface is the story

The guiding principle was simple: every UI element had to earn its place within the fiction. This isn't a website about Fahrenheit 451 — it's a website that exists inside the world of Fahrenheit 451. That distinction shaped every decision.

The black-dominant palette and degraded monospace typography weren't aesthetic choices — they were worldbuilding. They signal a system that operates in the margins, built by people who can't afford to be visible. The opening "WARNING: RESTRICTED AREA" modal doesn't just set the tone; it asks the user to commit to the premise before they can enter.

Status indicators borrowed from real system UIs — SYSTEM ONLINE, THREAT LEVEL: ELEVATED, ARCHIVE STATUS: SECURE — were deliberately designed to feel familiar and institutional, which makes them feel more unsettling in this context. The uncanny comes from recognising something real in a fictional frame.

CSS animations were restrained on purpose. Loading states, blinking cursors, and the `>` prompt prefix do more atmospheric work than a dozen scroll effects ever could.

Technical Approach

Astro + Supabase: a real stack for a fictional world

The brief called for a functional web experience — but I wanted to go further and build something that behaved like a real application, not just a static page dressed up to look dynamic.

Astro was the right framework for this: its component-based architecture let me build reusable "archive entry" and "classified document" components, while its SSG model kept the build fast and deployable to Netlify without a server. Critically, Astro's islands architecture meant interactive components only hydrated where needed — no unnecessary JavaScript overhead.

Supabase provided the PostgreSQL backend. Archive entries — the "preserved texts" — are stored in a real database and queried on page load, which means the archive genuinely grows when users register and submit content. This was my first time connecting a frontend project to a live backend service, and it fundamentally changed how I think about the data layer.

User authentication was implemented with Supabase Auth — users register a "keycard," log in, and gain access to the restricted archive. Session state persists across page navigation, with JavaScript managing route-level access control on the client side. The `LEVEL 1 CLEARANCE` login screen isn't decorative; it's a real auth gate.

Challenges

Where it got difficult

The hardest problem was synchronising Supabase's async data fetching with Astro's SSG pipeline. Astro pre-renders pages at build time — but the archive content is dynamic and user-generated, so it can't be known at build time. The solution was to use Astro's server-side rendering mode selectively for dynamic routes while keeping static generation for everything else, which required careful routing decisions early in the project.

Managing client-side session state without a framework like React was a deliberate constraint I set for myself. Every auth check, redirect, and conditional render was handled in vanilla JavaScript. It was tedious at times, but it forced me to understand what frameworks actually abstract away — and when that abstraction is genuinely worth it.

Getting the classified modal to feel right took longer than expected. The tension between theatricality and usability is real — too much animation and it becomes annoying; too little and the fiction falls flat. I settled on a single typewriter reveal with no looping effects.

Reflection

What I took away

01 · Learned
Backend integration changes your design thinking. Once the archive became a live database rather than static HTML, I had to consider empty states, loading states, error states, and edge cases I'd never thought about when building purely static work. UX and engineering are not separate disciplines.
02 · Would Do Differently
I'd define the data schema earlier. I refactored the Supabase table structure mid-project when I realised my original schema didn't support the filtering and classification system I wanted. Schema design is a design problem, not just a backend one.
03 · Proud Of
The narrative coherence. Every screen — from the onboarding modal to the empty-archive state ("The Vault is Empty. No texts have been preserved yet.") — speaks the same fictional language. The copy, the UI, the data model, and the visual design all tell the same story. That kind of consistency across layers is what I care most about.
